## [What M3 Adds to Your Codebase](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=what-m3-adds-to-your-codebase)

M1 built 5 services sharing one PostgreSQL database.  
M2 added 6 databases (polyglot persistence), authentication, caching, and design patterns — still one PostgreSQL, still cross-service SQL JOINs inside that PostgreSQL.  
M3 finishes the transformation:

-   **Database isolation** — each service gets its own PostgreSQL instance. No service can open a JDBC connection to another service's database.
-   **OpenFeign** — synchronous HTTP calls replace cross-service SQL JOINs for read dependencies.
-   **RabbitMQ** — asynchronous events replace cross-service write side-effects.
-   **Spring Cloud Gateway** — a 6th Maven module acts as the single entry point. JWT validation moves here.
-   **Kubernetes** — all services and databases deploy to a local MiniKube cluster.

### [What Does NOT Change](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=what-does-not-change)

-   All 45 M1 features — except the cross-service SQL inside ~22 of them (see sections below)
-   All 7 M2 design patterns
-   All 6 M2 databases (PostgreSQL + MongoDB + Redis + Elasticsearch + Neo4j + Cassandra)
-   JWT authentication (shared secret, stays the same)
-   Redis caching (all cached endpoints remain cached)
-   MongoDB event logging (Observer pattern stays in place)

### [New Booking Status Values](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=new-booking-status-values)

M3 adds saga-related statuses to the `Booking` entity's status enum. Existing M1 values (`PENDING`, `CONFIRMED`, `CHECKED_IN`, `COMPLETED`, `CANCELLED`) are preserved; new values are appended for saga lifecycle tracking:

> **Note on the M1 `COMPLETED` status:** M1 used `COMPLETED` as the booking's terminal state (set by S3-F4). In M3 the saga's natural terminal state is `PAID` (or `REFUNDED` on compensation). `COMPLETED` is retained in the enum for backward compatibility with M1/M2 reads but is no longer a saga waypoint — the saga path is `CHECKED_IN → COMPLETING → PAYMENT_PENDING → PAID/REFUNDED`.

## [Section 1 — Database Isolation](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=section-1-database-isolation)

### [1.1 What Changes](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_11-what-changes)

Every service previously connected to a single shared database:

```
spring:
  datasource:
    url: jdbc:postgresql://postgres:5432/eventticketingdb
```

In M3, each service connects to its own database on its own PostgreSQL instance:

```
# user-service application.yml
spring:
  datasource:
    url: jdbc:postgresql://user-postgres:5432/etdb-users

# event-service application.yml
spring:
  datasource:
    url: jdbc:postgresql://event-postgres:5432/etdb-events

# booking-service application.yml
spring:
  datasource:
    url: jdbc:postgresql://booking-postgres:5432/etdb-bookings

# ticket-service application.yml
spring:
  datasource:
    url: jdbc:postgresql://ticket-postgres:5432/etdb-tickets

# sales-service application.yml
spring:
  datasource:
    url: jdbc:postgresql://sales-postgres:5432/etdb-sales
```

> **Migration note:** the M2 database `eventticketingdb` is intentionally retired. M3 uses 5 fresh per-service databases (`etdb-users`, `etdb-events`, `etdb-bookings`, `etdb-tickets`, `etdb-sales`) — re-seed data per service after the new StatefulSets come up. Do not try to drop-and-recreate the M2 database under the same name.

### [1.2 Cross-Service FK Columns Become Plain Longs](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_12-cross-service-fk-columns-become-plain-longs)

Every `@ManyToOne` or `@JoinColumn` that pointed to another service's entity becomes a plain `Long` field. The column still exists in the database, but there is no JPA foreign-key relationship across databases.

> **Note:** Event Ticketing's M1 already stored cross-service references as plain `Long` columns (per M1 §7.7), so no JPA `@ManyToOne` relationships need to be removed. The M3 change is removing **cross-service SQL JOINs** that referenced these columns, not the columns themselves.

### [1.3 NoSQL Databases — Shared Instance, Separate Ownership](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_13-nosql-databases-shared-instance-separate-ownership)

MongoDB, Redis, Elasticsearch, Neo4j, and Cassandra remain as **single shared instances** (one StatefulSet each in Kubernetes). Each service already owns its own collections/indexes/keyspace and never reads another service's data — the logical isolation from M2 is sufficient. Running 5 MongoDB + 5 Redis + 5 Elasticsearch + 5 Neo4j + 5 Cassandra StatefulSets would make MiniKube unrunnable.

**The M3 rule:** No service connects to another service's PostgreSQL. Each service continues to own its MongoDB collection (`auth_events` / `event_events` / `booking_events` / `ticket_events` / `payment_audit_trail`), Redis key prefix, Elasticsearch index (S2's `event_search`), Neo4j label set (S3's UserNode/EventNode/ATTENDED), and Cassandra keyspace (S4's `ticket_scan_events`).

### [1.4 Deliverables for DB Isolation](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_14-deliverables-for-db-isolation)

-   `user-service/application.yml` — datasource URL points to `user-postgres:5432/etdb-users`
-   `event-service/application.yml` — datasource URL points to `event-postgres:5432/etdb-events`
-   `booking-service/application.yml` — datasource URL points to `booking-postgres:5432/etdb-bookings`
-   `ticket-service/application.yml` — datasource URL points to `ticket-postgres:5432/etdb-tickets`
-   `sales-service/application.yml` — datasource URL points to `sales-postgres:5432/etdb-sales`
-   All native-SQL JOINs that referenced other services' tables removed; behavior moved to Feign or RabbitMQ per §3–§8
-   New saga statuses added to `Booking.status` enum (`COMPLETING`, `PAYMENT_PENDING`, `PAID`, `PAYMENT_FAILED`, `REFUNDED`)

### [2.4 Error Handling](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_24-error-handling)

Wrap every Feign call in try-catch. Never let a downstream failure crash the calling service, for example:

```
try {
    BookingSummaryDTO summary = bookingServiceClient.getUserBookingSummary(userId);
    return buildDTO(user, summary);
} catch (FeignException.NotFound e) {
    return buildDTO(user, BookingSummaryDTO.empty());
} catch (FeignException e) {
    log.warn("booking-service unavailable for user {}: {}", userId, e.getMessage());
    throw new ServiceUnavailableException("Booking service temporarily unavailable");
}
```

### [2.7 Topology — What Each Service Must Declare](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_27-topology-what-each-service-must-declare)

Each service declares its own RabbitMQ topology as Spring `@Bean`s in a `@Configuration` class. The responsibilities are split:

-   **Producer service** declares only the `TopicExchange` it publishes to.
-   **Consumer service** declares the `Queue`, the `DLQ`, another `TopicExchange` reference (same name — Spring deduplicates), and the `Binding` that connects them.

Every consumer queue must have a dead-letter queue. The DLQ is wired by declaring the consumer queue with the `x-dead-letter-exchange` and `x-dead-letter-routing-key` arguments pointing at a separate dead-letter exchange + DLQ. Combined with `default-requeue-rejected: false` (§2.6), this means: when a listener method throws and Spring's retry exhausts (`max-attempts: 3`), the message is automatically routed to the DLQ — no manual `basicAck`/`basicNack` code in the consumer.

The exchange type for all Event Ticketing events is **`TopicExchange`**. This allows routing key wildcards so future events can be added without declaring a new exchange.

### [2.8 Event Payload Records](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_28-event-payload-records)

```
// ticket-service

public record TicketStatusChangedEvent(Long ticketId, Long bookingId, String newStatus) {}

// sales-service
public record PaymentInitiatedEvent(Long saleId, Long bookingId, BigDecimal amount) {}
public record PaymentCompletedEvent(Long saleId, Long bookingId, BigDecimal amount) {}
public record PaymentFailedEvent(Long saleId, Long bookingId, String reason) {}

// user-service

```

### [2.9 Consumer Idempotency Requirement](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_29-consumer-idempotency-requirement)

RabbitMQ delivery is at-least-once: with `acknowledge-mode: auto` + Spring's `max-attempts: 3` retry, a network blip or listener exception can replay the same message. Every consumer below must be idempotent — re-processing the same event must not double-count, double-create, or double-refund.

Required patterns per consumer type:

-   **Counter increments** (S1/S2 `booking.completed` and `booking.cancelled` stats updates) — track processed event IDs in a small `processed_events(event_id PRIMARY KEY, processed_at)` table per service. Skip if already processed; insert + mutate atomically inside `@Transactional`. Alternatively, derive stats on read (no mutating consumer at all).
-   **Create-once entities** (S5 `booking.completed` → create PENDING TicketSale) — add a `UNIQUE` constraint on `ticket_sales.booking_id` and use `ON CONFLICT DO NOTHING` semantics, or wrap in `findByBookingId().orElseGet(...)` so duplicate delivery is a no-op.
-   **State-guarded mutations** (S5 `booking.cancelled` → refund, S4 `booking.cancelled` → cancel tickets) — read current entity status before mutating; only act if the source state allows it (e.g., refund only if status IN (PENDING, COMPLETED) — already-REFUNDED is a no-op).
-   **Cross-status fan-out** (S3 consumers of `payment.*` events) — guard each transition; ignore the event if the booking has already advanced past the expected source status.

Treat any consumer that mutates state without one of these guards as a bug.

#### [\[S1-F3\] Get User Booking Summary](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=s1-f3-get-user-booking-summary)

**Branch:** `feat/M3/user/S1-F3/<studentID>`  
**Endpoint:** `GET /api/users/{id}/booking-summary`

**M1 implementation:** Direct native SQL JOIN on the shared `bookings` table using `user_id`, aggregating `totalSpent` from `bookings.totalAmount` where status = COMPLETED.

**M3 change:** Replace the SQL JOIN with a Feign call to booking-service:

```
@FeignClient(name = "booking-service", url = "${feign.booking-service.url}")
public interface BookingServiceClient {
    @GetMapping("/api/bookings/user/{userId}/summary")
    BookingSummaryDTO getUserBookingSummary(@PathVariable Long userId);
}
```

`BookingSummaryDTO` returned by booking-service: `{totalBookings, completedBookings, cancelledBookings, totalSpent, averageBookingAmount}`

user-service calls this and merges it with the local `User` data to build `UserBookingSummaryDTO` (with `userId`, `name`, plus the fields from BookingSummaryDTO).

**Test scenario:**

1.  (setup) In user-postgres: create User ID=1 (name="Ahmed Hassan"). In booking-postgres: create 5 bookings for userId=1 — 3 COMPLETED (amounts 300, 500, 700), 1 CANCELLED, 1 PENDING.
2.  (action) `GET /api/users/1/booking-summary` with valid Bearer token.
3.  (expect) 200 — `userId=1, name="Ahmed Hassan", totalBookings=5, completedBookings=3, cancelledBookings=1, totalSpent=1500.00, averageBookingAmount=500.00`.
4.  (verify) No direct JDBC connection from user-postgres to booking-postgres. The booking counts come from a Feign HTTP call.

#### [\[S1-F4\] Deactivate User Account](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=s1-f4-deactivate-user-account)

**Branch:** `feat/M3/user/S1-F4/<studentID>`  
**Endpoint:** `PUT /api/users/{id}/deactivate`

**M1 implementation:** `SELECT COUNT(*) FROM bookings WHERE user_id = ? AND status IN ('PENDING','CONFIRMED','CHECKED_IN')` directly on the shared database.

**M3 change:** Replace with a Feign call to booking-service `GET /api/bookings/user/{userId}/active-count` → returns `int`.

If the returned count > 0, throw 400 ("User has active bookings"). Otherwise set status = DEACTIVATED and save.

After deactivation: publish `user.deactivated` RabbitMQ event (see Section 8).

**Test scenario:**

1.  (setup) User ID=1 in user-postgres (status=ACTIVE). Booking in booking-postgres: userId=1, status=CONFIRMED.
2.  (action) `PUT /api/users/1/deactivate` → Feign → booking-service returns active-count=1.
3.  (expect) 400 — cannot deactivate user with active bookings.
4.  (setup) Update the booking status to COMPLETED in booking-postgres.
5.  (action) `PUT /api/users/1/deactivate` → Feign returns active-count=0.
6.  (expect) 200 — user status = DEACTIVATED in user-postgres. RabbitMQ `user.deactivated` event published.

#### [\[S1-F6\] Top Attendees by Spending](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=s1-f6-top-attendees-by-spending)

**Branch:** `feat/M3/user/S1-F6/<studentID>`  
**Endpoint:** `GET /api/users/reports/top-attendees?startDate={date}&endDate={date}&limit={n}`

**M1 implementation:** `JOIN users ON bookings.user_id = users.id GROUP BY users.id ORDER BY SUM(bookings.totalAmount) DESC` filtered by booking date range and status=COMPLETED.

**M3 change:** user-service cannot JOIN the `bookings` table (it lives in booking-postgres). Instead:

1.  Fetch all users from user-postgres.
2.  For each user, call Feign → `GET /api/bookings/user/{userId}/total?startDate=&endDate=` on booking-service → returns `BigDecimal` (total COMPLETED booking amount for this user in the date range).
3.  Sort users by total descending, take first `limit`.
4.  Build and return `List<TopAttendeeDTO>` with `{userId, name, totalSpent, bookingCount}`.

> **Note on date filtering:** Pass `startDate` and `endDate` as query params to the booking-service endpoint so it filters server-side rather than fetching all bookings.

```
@FeignClient(name = "booking-service", url = "${feign.booking-service.url}")
public interface BookingServiceClient {
    @GetMapping("/api/bookings/user/{userId}/total")
    BigDecimal getUserBookingTotal(
        @PathVariable Long userId,
        @RequestParam String startDate,
        @RequestParam String endDate
    );
}
```

**Test scenario:**

1.  (setup) 3 users in user-postgres. In booking-postgres: User A (userId=1) = 2000 total COMPLETED, User B (userId=2) = 5000 total, User C (userId=3) = 800 total, all within March 2026.
2.  (action) `GET /api/users/reports/top-attendees?startDate=2026-03-01&endDate=2026-03-31&limit=2`.
3.  (expect) 200 — `[{userId: 2, name: ..., totalSpent: 5000, bookingCount: ...}, {userId: 1, name: ..., totalSpent: 2000, bookingCount: ...}]`.
4.  (action) startDate=2026-04-01, endDate=2026-03-01 → 400.
5.  (verify) user-service made Feign calls to booking-service for each user's total. No direct query on `bookings` table.

#### [\[S1-F9\] Find Users by Favorite Category with Minimum Bookings](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=s1-f9-find-users-by-favorite-category-with-minimum-bookings)

**Branch:** `feat/M3/user/S1-F9/<studentID>`  
**Endpoint:** `GET /api/users/preferences/category?category={cat}&minBookings={n}`

1.  Query user-postgres for users whose `preferences->>'favoriteCategory'` matches the given value.

3.  (expect) 200 — only User A returned (User B has 2 completed bookings, below threshold).
4.  (action) `GET /api/users/preferences/category?category=&minBookings=1` → 400.
5.  (verify) Feign call made to booking-service for each candidate user.

#### [\[S4-F3\] Find Tickets Near an Event Venue](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=s4-f3-find-tickets-near-an-event-venue)

**Branch:** `feat/M3/ticket/S4-F3/<studentID>`  
**Endpoint:** `GET /api/tickets/nearby?lat={lat}&lon={lon}&radiusKm={r}`

**M1 implementation:** Three-table JOIN — `tickets JOIN bookings ON bookings.id = tickets.booking_id JOIN events ON events.id = bookings.event_id` — to read the event's venue coordinates from `Event.details` JSONB (`venueLat`, `venueLon`).

**M3 change:** Two-stage Feign chain replaces the JOIN.

1.  Query VALID tickets in ticket-postgres (local query).
2.  For each ticket, Feign → booking-service `GET /api/bookings/{bookingId}` to get the `eventId`.
3.  For each unique `eventId`, Feign → event-service `GET /api/events/{eventId}/venue-coords` → returns `{venueLat, venueLon}`.
4.  Compute Euclidean distance × 111, filter by radius, sort ascending, map to `NearbyTicketDTO` `{ticketId, attendeeName, bookingId, eventName, eventLat, eventLon, distanceKm}`.

> **Optimization:** Cache the `eventId → (lat, lon, name)` lookup in a request-scoped `Map` so repeated tickets for the same event do not re-call event-service. A batch endpoint `POST /api/events/batch-coords` on event-service would further reduce N HTTP calls to 1 — _future optimization, not required for M3_.

**Test scenario:**

1.  (setup) 3 events in event-postgres at different `venueLat/venueLon` JSONB values. 6 VALID tickets in ticket-postgres referencing bookings of those events.
2.  (action) `GET /api/tickets/nearby?lat=30.04&lon=31.23&radiusKm=5`.
3.  (expect) 200 — returns tickets for nearby events sorted by `distanceKm` ascending.
4.  (action) `radiusKm=0.001` → empty list (or only the closest ticket if within tolerance).

## [Section 8 — Booking Lifecycle Saga & Cancellation Cascade](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=section-8-booking-lifecycle-saga-amp-cancellation-cascade)

### [8.1 What Is a Choreography Saga](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_81-what-is-a-choreography-saga)

When a business transaction spans multiple services, there is no distributed rollback. The Choreography Saga achieves eventual consistency through:

1.  **Forward path:** each service listens for the previous step's success event and executes its part.
2.  **Compensation path:** on failure, the failing service publishes a failure event; every service that already committed reverses its local change on receipt of the compensation event.

> **Why "Booking Lifecycle Saga & Cancellation Cascade"?** Event Ticketing's domain flow is: an attendee books an event → tickets are issued → the attendee checks in → the event happens (CHECKED\_IN) → the booking is finalized (S3-F4 Complete) → the system records ticket-sale-of-record + payment. The saga trigger is therefore S3-F4 (`PUT /api/bookings/{id}/complete`), which in M1 already created the PENDING TicketSale as part of its multi-step transactional behavior. M3 promotes this to a Choreography Saga and adds payment-failure compensation that cancels the issued tickets and refunds the sale.

### [8.3 S3-F4 — Complete Booking (Saga Trigger)](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_83-s3-f4-complete-booking-saga-trigger)

**Branch:** `feat/M3/booking/S3-F4/<studentID>`  
**Endpoint:** `PUT /api/bookings/{id}/complete`

**M1 implementation:** Find booking. Validate status=CHECKED\_IN. Set status=COMPLETED. Calculate `totalAmount` (sum of `BookingItem.quantity × unitPrice`). Create a PENDING TicketSale record directly in the shared `ticket_sales` table.

**M3 change:** Remove the direct ticket-sale insert. Run three Feign pre-checks, then publish `booking.completed`.

Behavior:

1.  Find booking by ID → 404 if not found.
2.  Validate status = CHECKED\_IN → 400 if not.
3.  Calculate `totalAmount` if unset (sum of `bookingItems.quantity × unitPrice` from local `booking_items` table).
4.  **Pre-saga Feign checks** (all three must pass before any event is published):
    -   Feign → event-service `GET /api/events/{eventId}` → status must be `ONGOING` or `COMPLETED`; if 404 or status=UPCOMING/CANCELLED → 400 ("event has not yet occurred or was cancelled").
    -   Feign → user-service `GET /api/users/{userId}` → status must be `ACTIVE`; if 404 or DEACTIVATED → 400.
    -   Feign → ticket-service `GET /api/tickets/booking/{bookingId}/used-count` → returns `int`; must be ≥ 1; if 0 → 400 ("no attendance recorded for this booking").
5.  Mark booking status = `COMPLETING`, save.
6.  Publish `booking.completed` to `booking.events` exchange with payload `{bookingId, userId, eventId, totalAmount}`.
7.  Return 200 with the updated booking.

> Booking transitions from `COMPLETING` → `PAYMENT_PENDING` asynchronously when S3 consumes back the `payment.initiated` event from sales-service.

**Test scenario:**

1.  (setup) Booking ID=1 in booking-postgres: status=CHECKED\_IN, eventId=5, userId=10, items totaling 800. Event ID=5 in event-postgres: status=ONGOING. User ID=10 in user-postgres: status=ACTIVE. 4 Tickets in ticket-postgres for bookingId=1, 3 with status=USED.
2.  (action) `PUT /api/bookings/1/complete`.
3.  (expect) 200 — booking status=COMPLETING, totalAmount=800. `booking.completed` published to `booking.events`.
4.  (verify) No direct insert into `ticket_sales` table from booking-service.
5.  (action) Same booking, but ticket-service returns used-count=0 → 400. No event published.
6.  (action) User status=DEACTIVATED → Feign returns DEACTIVATED → 400. No event published.
7.  (action) Event status=UPCOMING → Feign returns UPCOMING → 400. No event published.
8.  (action) `PUT /api/bookings/999/complete` → 404 (booking not found). No Feign pre-checks issued; no event published.

### [8.4 S3-F7 — Cancel Booking](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_84-s3-f7-cancel-booking)

**Branch:** `feat/M3/booking/S3-F7/<studentID>`  
**Endpoint:** `PUT /api/bookings/{id}/cancel`

**M1 implementation:** Find booking. Validate status IN (PENDING, CONFIRMED). Set CANCELLED. `UPDATE tickets SET status='CANCELLED' WHERE booking_id = ? AND status='VALID'` directly on the shared database.

**M3 change:** Remove the direct `tickets` write. Publish `booking.cancelled` — ticket-service and sales-service consume it and handle their own side effects.

2.  Validate status IN (PENDING, CONFIRMED) → 400 if not.
3.  Set booking status = CANCELLED.
4.  Publish `booking.cancelled` to `booking.events` exchange with payload `{bookingId, userId, eventId, reason: "user_requested"}`.
5.  Return 200.

**Test scenario:**

2.  (action) `PUT /api/bookings/1/cancel`.
3.  (expect) 200 — booking status=CANCELLED. `booking.cancelled` published.

5.  (action) Try cancelling a COMPLETED booking → 400.
6.  (action) Try cancelling a CHECKED\_IN booking → 400.
7.  (action) `PUT /api/bookings/999/cancel` → 404 (booking not found). No event published.

### [9.3 JWT Global Filter](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_93-jwt-global-filter)

The JWT validation logic from M2 is reused in the gateway, but the filter wrapper must be rewritten. Spring Cloud Gateway is reactive (Project Reactor), so the M2 servlet `OncePerRequestFilter` cannot be reused as-is — `HttpServletRequest` and `HttpServletResponse` do not exist in WebFlux. Implement a `GlobalFilter` whose `filter(ServerWebExchange exchange, GatewayFilterChain chain)` returns `Mono<Void>` instead of `void`.

Header reads switch from `request.getHeader("Authorization")` to `exchange.getRequest().getHeaders().getFirst("Authorization")`. Forward enriched headers downstream by mutating the request:

```
ServerHttpRequest mutated = exchange.getRequest().mutate()
    .header("X-User-Id", uid)
    .header("X-User-Role", role)
    .header("X-Correlation-ID", correlationId)
    .build();
return chain.filter(exchange.mutate().request(mutated).build());
```

The JWT validation logic itself (parse, signature check, expiry check, claim extraction) reuses your M2 utility class unchanged — only the filter wrapper changes. Bypass `/api/auth/**` paths so register/login still work without a token.

Each service populates only the MDC keys relevant to its domain. `correlationId` is shared by all five services (set from the `X-Correlation-ID` header forwarded by api-gateway, or from the RabbitMQ message header in consumers). The remaining entity-specific keys differ:

#### [`logback-spring.xml`](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=logback-springxml)

The example below is the **booking-service** template (the busiest service — its JSON includes every entity field). Each other service uses the same XML structure but **drops the MDC fields it does not populate** from the `<message><pattern>` block.

```
<configuration>
    <appender name="LOKI" class="com.github.loki4j.logback.Loki4jAppender">
        <http>
            <url>http://loki.monitoring.svc.cluster.local:3100/loki/api/v1/push</url>
        </http>
        <format>
            <label>
                <pattern>app=eventticketing,service=${spring.application.name},level=%level,env=k8s</pattern>
            </label>
            <message>
                <pattern>
                    {
                      "timestamp": "%d{ISO8601}",
                      "level": "%level",
                      "service": "${spring.application.name}",
                      "thread": "%thread",
                      "logger": "%logger{36}",
                      "correlationId": "%X{correlationId:-}",
                      "userId": "%X{userId:-}",
                      "eventId": "%X{eventId:-}",
                      "bookingId": "%X{bookingId:-}",
                      "ticketId": "%X{ticketId:-}",
                      "saleId": "%X{saleId:-}",
                      "routingKey": "%X{routingKey:-}",
                      "message": "%msg"
                    }
                </pattern>
            </message>
        </format>
    </appender>
    <root level="INFO">
        <appender-ref ref="LOKI"/>
    </root>
</configuration>
```

#### [MDC Population](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=mdc-population)

-   **`correlationId`** — populated by a servlet filter (`OncePerRequestFilter`) that reads the `X-Correlation-ID` header set by api-gateway and calls `MDC.put("correlationId", value)`. The filter must clear MDC in `finally`. RabbitMQ consumers must also read the `correlationId` header from the inbound `Message` and call `MDC.put` at the start of the listener method.
-   **Entity IDs** (`userId`, `bookingId`, `eventId`, `ticketId`, `saleId`) — populated manually by service-layer methods using `MDC.put("bookingId", id.toString())` immediately before performing the operation, paired with `MDC.remove(...)` in a `finally` block to prevent leaking IDs into unrelated subsequent requests.
-   **`routingKey`** — set by RabbitMQ publishers and consumers to the routing key being processed (e.g., `booking.completed`, `payment.failed`). This makes the Layer 3 RabbitMQ event audit panel (§11.3) usable.

### [11.3 LogQL Panel Options (choose ≥ 3 per service)](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_113-logql-panel-options-choose-%e2%89%a5-3-per-service)

A LogQL query is built up in **three layers**, and every panel below uses all three:

1.  **Label** — `{app="eventticketing", service="booking-service", level="ERROR"}` — match log streams by the labels emitted by the Loki4J appender (§11.1). This narrows down which streams the rest of the query reads from.
2.  **Line** — `|= "search-text"`, `!= "exclude"`, `| json`, `| line_format "{{...}}"` — filter and parse individual log lines within the matched streams. Because messages are JSON (§11.1), `| json` exposes every field (`correlationId`, `bookingId`, `routingKey`, …) for further filtering.
3.  **Aggregator** — `count_over_time(...[1m])`, `rate(...[5m])`, `sum by (service) (...)` — turn the matching lines into time-series numbers that Grafana can plot.

#### [Available Panels](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=available-panels)

1.  **Error rate panel** — Count of ERROR-level log lines per service per minute.  
    _Example purpose:_ Spike detection — if booking-service logs 50 ERRORs in one minute, something is wrong.
    
2.  **Correlation ID trace panel** — Filter all log lines by a specific `X-Correlation-ID` value across all services.  
    _Example purpose:_ Trace a single booking creation request from api-gateway through booking-service, ticket-service, and sales-service.
    
3.  **RabbitMQ event audit panel** — Lines emitted by event publishers and consumers, filtered by routing key.  
    _Example purpose:_ Show how many `booking.completed` events were published vs. how many `payment.initiated` events were consumed in the last hour.
    
4.  **Feign call outcomes panel** — Log lines for successful Feign responses vs. `FeignException` catches.  
    _Example purpose:_ Detect when event-service is degraded — booking-service Feign calls to it start throwing exceptions.
    
5.  **Saga state transitions panel** — Log lines at each saga step filtered by bookingId.  
    _Example purpose:_ Visualize the complete saga flow for booking ID=42: COMPLETING → PAYMENT\_PENDING → PAID.
    
6.  **Slow operation warnings panel** — Log lines where elapsed time exceeded a threshold.  
    _Example purpose:_ Alert when S5-F10 tier analytics aggregation takes > 5 seconds.
    

___

### [11.4 PromQL Panel Options (choose ≥ 3 per service)](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_114-promql-panel-options-choose-%e2%89%a5-3-per-service)

A PromQL query is built up in **four layers**, and every panel below uses all four:

1.  **Metric** — the metric name itself, e.g., `http_server_requests_seconds_count` or `jvm_memory_used_bytes`. These are exposed by each service's `/actuator/prometheus` endpoint and scraped by Prometheus every 15s.
2.  **Label** — narrow the metric down with label matchers, e.g., `{service="booking-service", uri="/api/bookings", method="GET"}`. Labels come from Spring Boot's Actuator metrics and from the `job_name` set in `prometheus.yml`.
3.  **Range** — append a time window in square brackets, e.g., `[5m]` or `[1h]`. This turns the instant counter into a sequence of samples over that window so the function in layer 4 has data to operate on.
4.  **Function** — `rate(...)`, `increase(...)`, `histogram_quantile(0.99, ...)`, `sum by (uri) (...)`, `topk(5, ...)` — converts the range vector into the final per-second rate, percentile, top-N, or grouped aggregate that Grafana plots.

#### [Available Panels](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=available-panels-1)

1.  **HTTP request rate panel** — Requests per second per endpoint.  
    _Example purpose:_ Which booking-service endpoints are under the most load during peak hours?
    
2.  **HTTP latency percentiles panel** — P50/P95/P99 latency per endpoint.  
    _Example purpose:_ P99 latency on `GET /api/events/{id}/dashboard` is 4s — Feign enrichment is slow.
    
3.  **JVM health panel** — Heap usage, GC pause duration, thread count.  
    _Example purpose:_ Memory pressure before OOM — ticket-service heap at 90% after processing 10,000 scan events.
    
4.  **Database connection pool panel** — HikariCP active connections vs. pool size.  
    _Example purpose:_ Pool exhaustion alert — sales-service using 10/10 connections during saga fan-out.
    
5.  **Cache hit/miss ratio panel** — Redis cache hits vs. misses from `cache_gets_total`.  
    _Example purpose:_ S2-F12 dashboard cache hit rate — verify caching is effective.
    
6.  **RabbitMQ throughput panel** — Messages published vs. consumed per queue.  
    _Example purpose:_ Consumer lag on `payment.saga-listener` — published count > consumed count by > 100.
    

### [11.5 Observability Stack (K8s — monitoring namespace)](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_115-observability-stack-k8s-monitoring-namespace)

The three observability tools — **Loki**, **Prometheus**, and **Grafana** — run as their own pods inside the cluster, in a dedicated namespace called `monitoring`. Keeping them separate from the `eventticketing` namespace means observability resources (CPU, memory, restarts) are isolated from the application services, and an issue in the app does not take down the dashboards.

The data flow uses two opposite directions:

-   **Logs (push):** Each Spring Boot service runs the **Loki4J appender** (§11.1), which pushes log lines as JSON over HTTP to `http://loki.monitoring.svc.cluster.local:3100/loki/api/v1/push`. Loki itself never reaches into the services — they send to it.
-   **Metrics (pull):** **Prometheus** scrapes each service's `/actuator/prometheus` endpoint on a 15-second interval (configured below). "Scrape" here just means an HTTP GET — Prometheus pulls the current metric values from each service and stores them as time-series.
-   **Dashboards:** **Grafana** is configured with two datasources — Loki (for LogQL panels) and Prometheus (for PromQL panels). The 5 dashboard JSON files (one per service) are committed to the repo and provisioned into Grafana via a ConfigMap mount.

Cross-namespace DNS resolution makes this work: from `monitoring`, Prometheus reaches a service in `eventticketing` using the fully qualified name `<service-name>.eventticketing.svc.cluster.local`. From `eventticketing`, services push logs to `loki.monitoring.svc.cluster.local`.

#### [Two Namespaces Required](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=two-namespaces-required)

The cluster must contain **two namespaces**, each defined as its own YAML file under `k8s/namespaces/`:

```
# k8s/namespaces/monitoring-namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: monitoring
```

#### [Required Files & Directory Structure](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=required-files-amp-directory-structure)

All observability manifests live under `k8s/monitoring/`, separate from the application K8s tree shown in §10.1:

```
k8s/
├── namespaces/
│   ├── namespace.yaml                  # eventticketing (already in §10.1)
│   └── monitoring-namespace.yaml       # monitoring (new — see above)
```

#### [Required Manifests Per Component](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=required-manifests-per-component)

**Loki (StatefulSet):** Mount `loki-configmap` at `/etc/loki/`, attach the PVC at `/loki` for chunk storage. Service named `loki` so the Loki4J appender URL `http://loki.monitoring.svc.cluster.local:3100/loki/api/v1/push` resolves.

**Prometheus (Deployment):** Mount `prometheus-configmap` at `/etc/prometheus/prometheus.yml`. The ConfigMap holds the scrape config below. Attach the PVC at `/prometheus` for the TSDB.

**Grafana (Deployment):** Mount `grafana-datasources` at `/etc/grafana/provisioning/datasources/` and `grafana-dashboards` at `/etc/grafana/provisioning/dashboards/`. Service is `type: NodePort` on port 30030 so the dashboards are reachable from the host at `http://$(minikube ip):30030` (default credentials `admin/admin`, change on first login).

#### [Example Manifest — Prometheus Deployment](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=example-manifest-prometheus-deployment)

The full file at `k8s/monitoring/prometheus/prometheus-deployment.yaml`. Loki and Grafana follow the same pattern (different image, different mount paths, different ports — see "Required Manifests Per Component" above).

```
apiVersion: apps/v1
kind: Deployment
metadata:
  name: prometheus
  namespace: monitoring
spec:
  replicas: 1
  selector:
    matchLabels:
      app: prometheus
  template:
    metadata:
      labels:
        app: prometheus
    spec:
      containers:
        - name: prometheus
          image: prom/prometheus:v2.51.2
          args:
            - --config.file=/etc/prometheus/prometheus.yml
            - --storage.tsdb.path=/prometheus
          ports:
            - containerPort: 9090
          volumeMounts:
            - name: config
              mountPath: /etc/prometheus
            - name: storage
              mountPath: /prometheus
          readinessProbe:
            httpGet:
              path: /-/ready
              port: 9090
            initialDelaySeconds: 30
            periodSeconds: 10
      volumes:
        - name: config
          configMap:
            name: prometheus-config       # contains prometheus.yml — see scrape config below
        - name: storage
          persistentVolumeClaim:
            claimName: prometheus-pvc
```

The companion `prometheus-service.yaml` is a `ClusterIP` Service named `prometheus` exposing port 9090 — Grafana's Prometheus datasource uses `http://prometheus.monitoring.svc.cluster.local:9090` to reach it.

### [12.1 How Services Reference Files From Other Modules](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_121-how-services-reference-files-from-other-modules)

> **Package convention.** Every Java package and Maven `groupId` in this milestone follows the M1 team-based pattern `com.<teamID>.eventticketing.<service>` — replace `<teamID>` with your assigned team identifier (e.g., `com.team5.eventticketing.user`). Do NOT use the literal string `com.eventticketing.*`: it breaks the M1 grader's package detection (which expects a team prefix) and defeats plagiarism detection (every team would otherwise have identical package paths).

The `contracts/` module is the mechanism that lets all 5 services share Feign interfaces, DTOs, and event records **without duplicating any Java code**. It is a plain Maven JAR (no Spring Boot parent, no executable) that the 5 services depend on.

#### [How Java Code Imports Across Modules](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=how-java-code-imports-across-modules)

Example — booking-service calling event-service via Feign:

```
package com.<teamID>.eventticketing.booking.service;

import com.<teamID>.eventticketing.contracts.feign.EventServiceClient;       // from contracts module
import com.<teamID>.eventticketing.contracts.dto.EventDTO;                   // from contracts module
import com.<teamID>.eventticketing.booking.entity.Booking;                   // local to booking-service
import com.<teamID>.eventticketing.booking.repository.BookingRepository;     // local to booking-service

@Service
public class BookingConfirmationService {
    private final BookingRepository bookingRepository;
    private final EventServiceClient eventClient;

    public BookingConfirmationService(BookingRepository bookingRepository, EventServiceClient eventClient) {
        this.bookingRepository = bookingRepository;
        this.eventClient = eventClient;
    }

    public Booking confirm(Long bookingId, Long eventId) {
        EventDTO event = eventClient.getEvent(eventId);
        if (!"UPCOMING".equals(event.getStatus())) {
            throw new BadRequestException("Event is not upcoming");
        }
        // ... save booking with eventId, status=CONFIRMED ...
    }
}
```

Same pattern for events — sales-service consuming `booking.completed`:

```
package com.<teamID>.eventticketing.sales.messaging.consumers;
```

#### [Build Order — Maven Reactor Handles It Automatically](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=build-order-maven-reactor-handles-it-automatically)

Run `mvn clean install` from the repo root. Maven's reactor:

1.  Detects that the 5 services depend on `contracts:1.0.0`.
2.  Builds `contracts` **first**, installs it into `~/.m2/repository/com/<teamID>/eventticketing/contracts/1.0.0/`.
3.  Builds the 5 services in any order.
4.  Builds `api-gateway` last (or in parallel — it depends on neither contracts nor any service).

#### [Why This Eliminates Cross-Slice Compile Blockers](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=why-this-eliminates-cross-slice-compile-blockers)

When student A starts work on `S1-READ-DB` (which calls `BookingServiceClient.getUserBookingSummary(...)`), they need that interface to exist in `contracts/` so their code compiles. Day-0 kickoff (§13.2 Parallelism Strategy) ensures:

-   All 5 Feign client interfaces + all DTOs + all event records are committed to `contracts/` on **Day 0**, before any of the 15 slices begin work.
-   Student A's user-service compiles immediately because `BookingServiceClient` exists in the imported `contracts` JAR — even though student G (owner of `S3-READ-DB`) hasn't yet implemented the matching `GET /api/bookings/user/{userId}/summary` endpoint inside booking-service.
-   Runtime testing: student A uses `@MockBean BookingServiceClient` until student G's branch merges.

### [13.2 The 15 Deliverables](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_132-the-15-deliverables)

> **Rule:** Each deliverable is a **vertical slice** that touches **all parts** of M3 — Java code, Kubernetes manifests, and observability artifacts. No deliverable is purely Java, K8s, or YAML. The 15 slices are designed so every team member works in **parallel without blocking anyone else** (see "Parallelism Strategy" below the table).

The 15 deliverables are organized as **5 services × 3 vertical slices per service**:

-   **Slice A — Read & DB:** DB isolation, outbound Feign clients, exposed read endpoints, Postgres K8s, ≥3 LogQL panels, Logback config.
-   **Slice B — Events & Saga:** RabbitMQ topology, publishers/consumers, saga participation, Spring Boot K8s, ≥3 PromQL panels, actuator config.
-   **Slice C — Cross-Cutting Infra:** that service's gateway route entry + scrape job entry + dashboard JSON aggregation, plus **one** assigned shared-infra item.

#### [Parallelism Strategy — How All 15 Members Work Without Blocking Each Other](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=parallelism-strategy-how-all-15-members-work-without-blocking-each-other)

The 15 slices are designed so nobody waits for anyone else. The key is **contract-first development**: every cross-service interface is agreed in a kickoff meeting on Day 1, written down, and committed before any feature work starts. From that moment, each member writes against the contract — not against another member's implementation — so they can compile, test (with mocks), and deploy their slice independently.

1.  **Day-0 kickoff contracts (committed by the team lead, ~2 hours):**
    
    -   **Feign client interfaces** — every `@FeignClient` interface signature (e.g., `BookingServiceClient.getUserBookingSummary`) and the DTOs they return (`BookingSummaryDTO`, `EventBookingRevenueDTO`, etc.). Committed once to a `contracts/` Maven module that all services depend on.
    -   **Event payload records** — every `record` class (`BookingCompletedEvent`, `PaymentFailedEvent`, …) is added to that same `contracts/` module. Routing keys + exchange names are fixed in §2.10 (no team debate).
    -   **New endpoint paths + DTO shapes** — exact path, query params, response JSON. Already documented in each service's "New Endpoints" table (§3–§7).
    -   **K8s Service names** — `loki`, `prometheus`, `rabbitmq`, `<svc>-postgres` — fixed up-front so DNS resolves correctly across slices.
    -   **Shared YAML stub files** — `api-gateway/application.yml` (with route placeholders), `prometheus-configmap.yaml` (with scrape-job placeholders), `grafana-dashboards.yaml` ConfigMap (referencing 5 dashboard JSON paths). Each "INFRA" slice owns the _creation_ of one stub; each service slice fills in its own block.
2.  **Compile-time independence** — once the `contracts/` module is in place, slice 1's `BookingServiceClient.getUserBookingSummary(...)` call compiles even if slice 7 hasn't implemented `GET /api/bookings/user/{userId}/summary` yet. The interface is the only thing slice 1 needs to compile and unit-test.
    
3.  **Runtime independence (mocking)** — for local dev each slice uses `@MockBean` on Feign clients and Testcontainers RabbitMQ. A slice can run, deploy, and verify in isolation without the other 14 slices being merged.
    
4.  **Disjoint file ownership** — each slice writes to its own packages and YAML blocks. The only shared YAML files are `api-gateway/application.yml`, `prometheus.yml`, and `grafana-dashboards.yaml` — these have a stable structure agreed at kickoff so each slice edits only its assigned block. Merge conflicts are minimized to non-existent.
    
5.  **Deploy-time independence** — when a slice's branch is ready, it merges into `main` whenever; the merge order is **not** prescribed because no slice depends on another slice being merged first. Integration verification (saga end-to-end, gateway routing) happens after all 15 are merged, owned by `S5-INFRA`.

### [13.4 Merge Order](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=_134-merge-order)

Because the contract-first design eliminates compile-time dependencies, **merge order is unconstrained** — branches can be merged in any order, as long as the `contracts/` module exists in `main` first.

1.  **Day 0:** Team lead merges the `contracts/` Maven module + the 3 stub YAML files (`api-gateway/application.yml`, `prometheus-configmap.yaml`, `grafana-dashboards.yaml`) into `main`.
2.  **Day 1 onwards:** All 15 slices proceed in parallel; each merges to `main` when ready. No slice blocks another.

## [Section 16 — Critical Rules](https://scalable-docs.netlify.app/#/event-ticketing-m3?id=section-16-critical-rules)

1.  **No cross-service JDBC.** After M3, no service opens a JDBC connection to another service's database. Zero tolerance.
2.  **Feign for reads. RabbitMQ for side-effects.** Use Feign when you need data to continue processing. Use RabbitMQ when triggering a state change in another service.
3.  **Auto ACK with DLQ routing.** Use Spring's default `acknowledge-mode: auto` with `default-requeue-rejected: false`. Spring ACKs the message when the listener method returns normally and rejects when it throws — after retries are exhausted, rejected messages flow to the DLQ via the queue's `x-dead-letter-exchange` argument (no manual `basicAck`/`basicNack` calls).
4.  **DLQ for every queue.** Every consumer queue has a dead-letter queue. Failed messages are never silently dropped.
5.  **StatefulSet for all databases.** Never use plain `Deployment` for a stateful database.
6.  **Explicit constructor injection.** Consistent with M1/M2.
